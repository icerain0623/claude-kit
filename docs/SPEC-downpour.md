# downpour — スペック(L2)

## 概要

レールの**任意ビルド加速装置**。squall が作った `tasks.md`(ビルド台帳)を入力に、明示呼び出し(または monsoon の提案)で起動し、依存グラフからウェーブ実行計画を提示 → GO 一回で自律走行に入り、台帳の未ブロックタスクをサブエージェントで燃やし切る。Superpowers の subagent-driven development の長所(タスクごとの新鮮なコンテキスト・採点者の分離)を、レールのトレーサビリティ(機能 ID → タスク → EARS 完了条件)の上で動かす。

**downpour は新しい実行エンジンではない。** オーケストレータ=メインループの Claude 自身であり、スキルが与えるのは規律(台帳の書き手を一本化・ウェーブ計画・二段検証・停止条件)。価値は機械ではなく手順の一貫性にある。(Round 1 #2 の疑問への回答として記録)

## 決定事項 — Phase 0

### D1. 立ち位置 — 任意の加速装置(標準駅ではない)
デフォルトのビルドは通常ループ(対話的)。台帳に自律で回せるまとまりがあるとき `/downpour` を明示起動。monsoon は「未ブロックタスク3件以上」で**提案**(強制しない)。標準駅への昇格は v2 以降で再検討。

### D2. 実行単位 — ウェーブ燃やし切り+開始前 GO 一回
依存グラフの「今すぐ着手可能な層」を実行し、ブロックが外れた次のウェーブへ続行。開始前に実行計画(ウェーブ構成・並列/直列の判定根拠・停止条件)を1画面提示して GO 一回。停止条件: 全完了 / 全ブロック / 失敗バジェット到達 / 指定範囲到達。

### D3. 並列方式 — ハイブリッド(ファイルフットプリント判定)
触りそうなファイルが重ならないタスクだけ同一 worktree 内で並列、重なる可能性があれば直列。判定根拠は GO 画面に明示。worktree 分離並列は v2。

### D4. 検証設計 — 二段(タスクごと)+品質はウェーブ末一括
①実装サブエージェント(`check` 込み)→ ②別コンテキストの検証エージェントが EARS 完了条件だけを判定。NG は理由つき差し戻し1回 → 再 NG で保留+失敗1件。品質レビューはウェーブ末に一括。

## 決定事項 — Round 1(実行詳細)

- **E1. 文脈渡し**: タスク行(ID/タイトル/完了条件/依存)+該当機能 ID の spec 抜粋+設計セクションへの**パス参照**(全文は渡さない)。規約は repo CLAUDE.md の自動読込に任せる。
- **E2. 台帳書き戻し**: オーケストレータのみが書く。状態欄+`## 進捗ログ` へ `T-003 done (downpour, <branch>, <sha>) 検証: PASS` 形式で追記。書込前に再読、外部介入を検知したら書かずに停止・報告。
- **E3. コミット規約**: タスクごとに1コミット。タイトルは自然な conventional commits(`<type>(<scope>): <題>`)、**タスク ID は本文に `Task: T-003`** で記録(タイトルへの強制はしない — Round 1 #3 でユーザー調整)。ウェーブ末レビュー修正は別コミット。push はしない。
- **E4. 差し戻し・保留**: 検証 NG の理由(どの EARS 節が・どう満たされないか・確認手順)を**要約せずそのまま**実装エージェントへ。保留は台帳行に `保留(理由1行)`+詳細は進捗ログ。保留の下流はブロック扱い。
- **E5. ウェーブ末レビュー**: code-review 相当(medium)。確信度の高い修正のみ自動適用(独立コミット)、不確実な指摘は報告へ。behavior 変更は禁止。
- **E6. monsoon 連携**: 未ブロック todo 3件以上 かつ express-lane 作業中でない → 提案(Decision リストに1行追加)。自動起動はしない。
- **E7. 呼び出し**: slash-only(`disable-model-invocation: true`)。引数で範囲指定(`T-005まで` / `3タスク` / 無指定=停止条件まで)。
- **E8. 並列・model/effort**: 同時最大3。実装=セッション model 継承、検証=同 model・低 effort、ウェーブ末レビュー=medium。
- **E9. バジェット**: 失敗バジェット デフォルト2件(差し戻し1回で復帰は数えない)。トークン上限なし、ウェーブ境界で消費報告。
- **E10. エスカレーション(二段)**: 仕様の穴が**そのタスク限り** → 保留+feedback.md に Open question+続行。**後続の作業が続行不能・手戻り増大の見込み** → 実行を停止してユーザーに質問を提示(Round 1 #10 でユーザー調整)。
- **E11. v1 の線**: v1 = D1〜D4+E1〜E10。v2 = worktree 分離並列 / 標準駅昇格の再検討 / 失敗パターンの sunbreak 連携。保留 = リモート実行。
- **E12. 概算**: スキル本体の実装は S〜M(SKILL.md 1本+monsoon 1行+README/ガイド追記。新規コードなし — 手順の符号化)。

## 決定事項 — Round 2(fresh-eyes レビュー 15 件の解決)

- **R1. コミットプロトコル**: サブエージェントはコミットしない(編集のみ)。完了報告後、**オーケストレータが当該タスクのフットプリントのパスだけを stage し、タスク完了順に直列コミット** — index 競合と巻き込みコミットを構造的に防ぐ。
- **R2. フットプリントの入力源**: 第一入力 = tasks.md の任意列「対象パス」(squall 側スキーマに任意列として追記)。無ければ設計 docs+spec から推定し、GO 画面に推定である旨を明記。
- **R3. 台帳スキーマ**: squall `detail-design-jp.md` セクション7(タスク分解)を参照。最小前提: タスク行 = ID / 依存 / 完了条件 / 状態(todo|in-progress|done|保留)、`## 進捗ログ` は append-only。
- **R4. エージェント自体の失敗**: 実装エージェントの失敗(クラッシュ / check 不通過 / 成果なし)= 検証 NG と同等(差し戻し1回 → 保留+失敗1件)。検証エージェントの失敗(エラー / 判定不能)= 1回再実行 → 再失敗で保留+失敗1件(実装側の責でない旨を進捗ログに記す)。**ユーザーの判断・操作なしに解決できない状況では停止してユーザーに通知する**(可能なら push 通知、なければ停止報告を目立たせる)。
- **R5. 検証エージェントの入力**: ①タスクの EARS 完了条件 ②当該タスクの変更 diff ③コマンド実行権限。実装エージェントの経緯・言い分は見せない(採点の独立性)。
- **R6. GO の拒否・修正**: NO → 何も実行せず終了報告。修正指示 → 計画を再構成して再提示、GO まで実行しない。
- **R7. ウェーブ途中の停止**: 停止条件成立時は新規投入を止め、実行中エージェントは完走を待ち、完走分は通常どおり検証・コミット・記録してから停止報告。
- **R8. 保留カウント**: 失敗カウント = 検証由来の保留のみ。仕様穴由来の保留は数えない(E10 の続行意図を保つ)。
- **R9. 低重大度の確定**: (a) 失敗バジェットは v1 固定2件 (b) 範囲指定は完了タスク数で数え、指定タスク到達不能は「全ブロック」扱い (c) 外部変更検知 = 直近の自書き全文スナップショットとの比較 (d) 起動時プリフライト(台帳存在・クリーンツリー・非 main ブランチ)不成立なら計画を出さず報告 (e) ウェーブ末修正コミット後に `check` 再実行、失敗なら revert。

## 受け入れ条件(v1・EARS)

- **AC-1**: 未ブロックタスクが無いとき、実行計画を出さずにその旨と原因(全完了 or 全ブロック)を報告する
- **AC-2**: 実行開始前に、ウェーブ構成・並列/直列の判定根拠・停止条件を含む実行計画を提示し、GO を得るまで実行しない
- **AC-3**: タスクが完了したとき、台帳の状態欄と進捗ログが同一 invocation 内で更新されている
- **AC-4**: 検証エージェントが NG を出したとき、NG 理由つきで実装エージェントに1回だけ差し戻す
- **AC-5**: 同一タスクが再度 NG になったとき、保留マーク+失敗カウント+次の未ブロックタスクへ進む
- **AC-6**: 失敗カウントがバジェットに達したとき、実行を停止し、完了/保留/未着手の内訳を報告する
- **AC-7**: 台帳が invocation 中に外部から変更されていたとき、書き込まずに停止して報告する
- **AC-8**: 仕様の穴が**未着手タスクの依存経路上にある**(=下流をブロックする)とき、実行を停止してユーザーに質問を提示する
- **AC-9**: 起動時プリフライト(台帳が存在しパース可能 / 作業ツリーがクリーン / カレントブランチが main でない)が不成立のとき、実行計画を出さずにその旨を報告する

## 追補 — 実装レビューによる修正(2026-07-03)

実装直後の code-review(8ファインダー+検証)で確定した仕様修正。上記本文より本節が優先する:

- **check の実行(E8/D4 修正)**: 直列タスクは実装エージェント自身が回す。並列タスクではエージェントは回さず、バッチ完走後にオーケストレータが一回。失敗は変更パスで帰属先を特定(帰属不能ならウェーブ残を直列化)。
- **in-progress の書き手(R3 補完)**: タスク投入時にオーケストレータが状態欄を in-progress に更新(台帳書込プロトコルに従う。進捗ログ行は不要)。
- **コミットの stage 基準(R1 修正)**: フットプリントではなく**実変更パス**。並列時、見積もり外の編集は兄弟と衝突しなければ当該タスクとして stage+進捗ログに記録、同一ファイルを兄弟と触っていたら NG 相当+ウェーブ残の直列化。検証 diff も実変更パスで切り出す。
- **AC-7 の判定基準(R9c 修正)**: プリフライト読取の内容ハッシュを起点、自書きごとに更新、書込前にハッシュ照合。全文スナップショットの保持はしない。
- **差し戻し回数(AC-4 明確化)**: タスクごと通算1回(原因を問わない)。
- **AC-8 と E10 の境界(明確化)**: 穴を踏んだタスクに未着手の依存下流がいなければ続行、いれば停止して質問 — 依存グラフで機械的に判定。
- **ウェーブ末 revert(R9e 明確化)**: `git revert`(履歴を書き換えない)。適用→revert は最終報告に記載。
