# 要件定義モード（L3）テンプレート

petrichor が **L3（要件定義）** を選んだときに従う成果物体系。実務の要件定義・QA 知見に基づく
（業務要件 / 機能要件 / データ設計 / CMS / 非機能要件 / システム構成）。
**成果物の本文は spec 言語設定に従う**（`00-overview.md` ヘッダに lang があればそれ、なければ ja。`ja+en` は正本＋Done 時レンダリング訳 — 並行執筆しない）。この体系を Phase 1+ の topics（＝カバレッジ・チェックリスト）に使う。

## スコープ境界（要件定義はどこからどこまでか）

- petrichor が扱うのは「**何を作るか**」= 要件定義 〜 **基本設計（外部設計）** まで。
- 「**どう作るか**」= 詳細設計（モジュール設計・API 内部・シーケンス図・インフラ設定値）、実装、テスト、
  納品、運用は **後工程＝petrichor の外**（実装ループ / `squall` 以降へ渡す）。
- 上流の **ヒアリング**（アジェンダ / 議事録）と **RFP** は要件定義の入力。petrichor では **Phase 0** で集める。
- クライアント向けの **提案書**（見積・体制・契約条件）は案件による。個人 / 社内プロジェクトでは任意。

## 非 Web プロジェクトの読み替え

セクション構成は Web 制作の形（画面・CMS・帳票）で書かれているが、個人開発では CLI / ライブラリ / ゲーム等も来る。
B 群は次のように**読み替えて適用**し、読み替えた旨をセクション冒頭に記録する（黙って飛ばさない）:

- **CLI / TUI**: 画面一覧 → コマンド・サブコマンド一覧（引数 / オプション / 終了コード）、
  画面定義書 → 入出力定義（stdin/stdout/stderr・エラーメッセージ・対話プロンプト）。
- **ライブラリ / SDK / API サーバ**: 画面一覧 → 公開 API（エンドポイント / 関数）一覧、
  画面定義書 → シグネチャ＋入出力＋エラー仕様＋使用例。
- **ゲーム**: 画面一覧 → シーン / メニュー一覧、業務フロー → ゲームループ / プレイヤージャーニー。

読み替え先が存在しないセクションは通常どおりスキップ（理由を記録）。

## 全体像と依存関係（この順で「根拠」が前へ流れる）

業務要件定義 → 機能要件定義 → 非機能要件定義 → システム構成 →（運用計画・プロジェクト計画＝見積 / スケジュール）

- 各成果物は前段を「根拠」にする。前段が **5〜8 割**固まれば次に着手してよい（100% 待たない）。

## 横断原則（全セクション共通の振る舞い）

- **5〜8 割の確度で着手し、穴埋めしながら完成**（部分回答 OK・未確定は次ラウンドへ繰り越し）。
- **全要素に ID を振り**、後続成果物の ID を追記して**トレース**する
  （要求 ID → 業務 ID → 機能 ID → 画面 ID → データ …）。落とし込み漏れを防ぐ。
  ID は見出しにし（`## REQ-LIB-01 …`）、**言語に依存させない**（`ja+en` のレンダリング訳でも同一 ID —
  訳してもトレースと各版内のリンクが生き残る）。他成果物からの参照は `[[ファイル#ID]]` の Obsidian リンクで書く
  — クリックで定義へ飛べ、バックリンクで「この ID に触れた決定・機能・画面」が自動で逆引きできる。
  **リンクを付けるだけに留める**：ID ごとのノート化や、block ref での行単位アンカー、トレース表の自動生成までは
  作り込まない（ファイルが増え、人も AI も読みづらくなる）。pin-point が要るときも見出し単位のリンクで止める。
- **As-Is（現状）と To-Be（理想）を必ず区別**する。
- 業務は **5W1H**（誰が / いつ / どこで / 何のために / 何を / どうする）。
  HOW（どうする＝実現方針）は仮で入れ、機能要件で確定する。
- 機能は **CRUD** を意識し、**例外・状態変化**、そして「**画面のない機能**」（通知 / バッチ / 権限制御）を漏らさない。
- 「**業務 → 機能 → 画面 → 操作 → データ**」が一本の線で説明できること。
- 非機能は定性でなく **定量**（数値＋測定方法＋判断基準）。必要なら **SLA** で合意。
- **スコープの線引き**: 機能要件一覧の各機能に **優先度（v1 / v2 / 保留）** と **概算（S / M / L）** を振り、
  「**v1 の線**」を引く。個人開発で最大のリスクは曖昧さより**盛りすぎ** — v1 だけで目的が最低限成立する
  ことを Done の条件にする（概算タグがあると線引きがコストの判断になる）。
- **受け入れ条件**: 各機能 ID に**検証可能な受け入れ条件**を 1 行以上。推奨文型は **EARS 記法** —
  「**〜のとき（WHEN）、システムは〜する（SHALL）**」、例外は「**もし〜なら（IF）、…**」。
  文型の時点で条件・主体・挙動が揃うので、検証可能性が構文で担保される（「使いやすいこと」のような
  文は EARS に直せない＝要件として未成熟のサイン）。後工程の出典になる —
  `squall` の `tasks.md` はタスク完了条件をここから導出し、実装後の `verify` はここに照らして挙動を確認する
  （仕様を write-once にしない）。
- 各成果物のゴールは「**見積算出が可能な粒度**」。
- 案件に該当しないセクション（CMS / 帳票 / バッチ / 外部連携 / 提案書 など）は **スキップしてよい**が、
  スキップした旨と理由を記録する（黙って飛ばさない）。

## セクション（成果物）一覧

各セクションは **目的 / 開始条件 / 終了条件(Done) / 主な項目 / レビュー観点** を持つ。
Phase 1+ では開始条件の依存順に進め、終了条件＋レビュー観点を満たしたら「確定」にする。

### A. 業務要件系

**A-1. 要求・前提・制約条件一覧【必須】**
- 目的: ヒアリング / RFP の全要望・前提・制約を集約し一覧化（ID 管理）。
- 開始条件: ヒアリング議事録・RFP 等の入力がある。
- 終了条件: 全ての要望 / 前提 / 制約が漏れなく ID 付きで記載され、出典（いつ・誰）が明記され、実現方針欄がある（仮でも可）。
- 主な項目: 要求 ID / カテゴリ / 概要 / 背景 / 出典 / 実現方針（随時確定）。
- レビュー観点: 漏れなし / 出典が明確 / 文脈に依存しない表現 / ID 付与 /「漏れなく要件定義できたか」のチェックシートとして使えるか。

**A-2. 業務一覧・業務権限一覧【必須】**
- 目的: 運用開始後に発生する全「業務」（人・システム問わず）を 5W1H で洗い出し、権限（誰が何をできるか）を整理。
- 開始条件: A-1 がある。
- 終了条件: 5W が明確 / A-1 の内容を反映 / ビジネス成立に必要な業務に漏れなし（対になる業務が揃う）/ イレギュラー考慮 / 権限マトリクスがある。
- 主な項目: 業務 ID / カテゴリ / 業務名 / 概要(5W1H) / As-Is / To-Be(実現方針→機能 ID) ／ 権限: ロール × 業務の可否。
- レビュー観点: 5W 明確 / 漏れ業務なし / イレギュラーが考慮されている（しない場合はその旨記載）。

**A-3. 業務フロー図【任意】**
- 目的: 複数ステークホルダー / システム連携のある業務の流れを図解（As-Is / To-Be）。
- 主な項目: 主語（業務の主体者）＋述語（時系列のアクション）。

### B. 機能要件系（機能・外部設計）

**B-1. 機能要件一覧【必須】**
- 目的: システムで実装する機能の全量を一覧化（業務要件と画面設計の中間成果物）。
- 開始条件: A 群が 6〜7 割整理されている。
- 終了条件: 必要機能が漏れなく / 各機能で「何のため・誰が使う・何ができる」が言える / 見積可能な粒度 / **v1 の線が引かれている**。
- 主な項目: 大分類 / 中分類 / 機能 / 内容 / 権限（ロール別）/ 対応業務 ID / **優先度（v1/v2/保留）** / **概算（S/M/L）** / **受け入れ条件** / 設計検討事項。
- レビュー観点: 業務との対応が説明できる / 粒度が実装判断可（「管理できる」等の抽象で止まらない）/ 画面のない機能（通知・権限・状態・例外）も含む / 権限差分が機能として定義されている / **v1 だけで目的が最低限成立する（全部 v1 は線引きの放棄）** / **受け入れ条件が検証可能な文になっている（実装後に verify で確認できる）**。

**B-2. 画面一覧【必須（画面のある案件）】**
- 目的: 提供する画面の全量を整理（機能要件と画面設計の橋渡し）。
- 主な項目: 画面名 / Path / 機能 ID / 機能分類 / 権限別の表示。
- レビュー観点: 機能が必ずどこかの画面に紐づく / 画面の役割が明確（一覧 / 詳細 / 入力）/ 遷移が想像できる。

**B-3. 画面定義書【L3・画面のある案件】**
- 目的: 画面ごとの表示 / 操作 / 振る舞いをユーザー視点で定義（シナリオテストにつながる）。
- 主な項目: 項目 / 属性 / 入力可否 / 桁・型・制約 / アクション・イベント / 処理概要 / 入力・出力 / 遷移先。
- レビュー観点: 操作が一意に決まる / 業務ルールが UI の振る舞いに落ちている / エラー時・未入力時の挙動が定義済み（実装者の解釈任せにしない）。

**B-4. 帳票一覧 / 設計書【条件付き（帳票がある場合）】**
**B-5. バッチ一覧 / 設計書【条件付き（バッチ処理がある場合）】**
**B-6. 外部連携一覧 / 設計書【条件付き（外部システム連携がある場合）】**
- 各: 一覧＋ IF 概要。外部連携 ID でトレースする。

### C. 機能要件系（データ）

**C-1. データモデル図（ER 図・概念 / 論理）【必須】**
- 目的: システム全体のデータ構造を俯瞰する。
- 開始条件: 機能一覧が確定、画面一覧・画面設計が一通り揃っている。
- レビュー観点: 機能・画面で必要なデータを網羅 / 不自然な多対多が乱発されていない / マスタとトランザクションが適切に分離。
- ヒント: Mermaid で書くとメンテナンスが楽。

**C-2. データフロー図（DFD）【推奨】**
- 目的: データがどこで作られ / 更新され / 参照されるかを可視化。
- レビュー観点: 発生点（画面・API・バッチ）が明確 / 更新タイミングが複数化していない / 同義データを複数箇所で持っていない。

**C-3. データ項目定義書【必須（L3）】**
- 目的: データ設計の最終成果物。実装・テスト・運用の共通言語。
- 終了条件: 全ての画面入力・表示項目がどのテーブル / カラムに保存されるか説明可 / 作成・更新・参照の流れが追える / 各項目の意味・制約が文章で説明できる。
- 主な項目: 物理名 / 論理名 / 説明 / PK・FK / NOT NULL / データ型 / 桁数 / 制約 / 例。
- レビュー観点: 画面 → データの落とし方が妥当 / 拡張を見越した分離（マスタ・履歴）/「実装者が推測しないと分からない定義」は NG / 画面都合のテーブル乱造・JSON 逃げ・履歴の後付けは NG。

### E. 非機能要件系

**E-1. 非機能要件一覧【必須】**
- 目的: 「どのように動くか」=品質・性質を定義（IPA 非機能要求グレード準拠で抜け漏れ防止）。
- 開始条件: 機能要件一覧があり、主要画面・操作フロー、利用者・利用環境（想定ユーザー数 / 地域 / 端末）が整理されている。
- 終了条件: 主要カテゴリ（可用性 / 性能・拡張性 / 運用性・保守性 / 移行性 / セキュリティ / システム環境）が定義され、**定量基準＋測定方法**があり、品質について合意できている。
- ヒアリング項目例: 可用性（稼働時間・計画停止・RTO / RPO）/ 性能（同時接続数・最大応答時間・バッチ処理時間）/ セキュリティ（認証方式・暗号化・ログ保持 / 監査証跡）/ 拡張性（将来のユーザー増・データ量増への耐性）。
- SLA（必要時）: 可用性(%) / 平均応答時間 / 障害検知から復旧までの目標時間、および SLA 違反時の取り扱い。
- レビュー観点: 各項目が定量化 / 測定方法が明確 / SLA 保証範囲が明示 / テスト可能な状態 / 例外・制約（計画停止等）が合意されている。

**E-2. システム運用業務一覧【推奨】**
- 目的: 運用フェーズで発生する業務（監視・バックアップ・障害対応など）を要件として洗い出す。

### F. システム構成系

**F-1. システム構成図（As-Is / To-Be）【必須】**
- 目的: 構成要素とネットワーク / 相互関係を図示。機能要件・非機能要件の**両方**が影響する。
- 開始条件: 現行構成（あれば）が判明 / 利用技術（サービス・言語）が決定 / 業務一覧・機能一覧が確定 / 非機能要件が確定。
- 終了条件: 構成要素が漏れなく記載 / アクセス経路・データの流れが理解可 / 要件を実現できる環境である / インフラ担当とアプリ担当の認識が一致 / 調達・運用体制が決定 / 構築・運用費が見積可能な粒度。
- ヒント: 編集しやすいデータで作成（後フェーズのリリース計画 / 運用マニュアルでも活用）。本番だけでなく STG も。

### G. 統合・計画

**G-1. 要件定義書（最上位・統合）**
- 全ての要件を統合した最上位ドキュメント。petrichor では `00-overview.md` がこれに相当する。

**G-2. プロジェクト計画（見積 / スケジュール / 体制）【案件による】**
- 上記成果物を根拠に、開発・運用の見積とスケジュールを置く。

## L3 全体の完了条件

該当する全セクションが各「終了条件」を満たし、相互の ID 整合が取れ、**v1 の線が引かれ**、
各機能に検証可能な**受け入れ条件**があり、
ステークホルダー（個人案件なら本人）が「**この内容で設計・実装に進んでよい**」と判断できる状態。
