# 詳細設計（squall）テンプレート

squall が従う詳細設計の成果物体系。実務の詳細設計・QA 知見に基づく（詳細設計1・2 / 基本設計 / 運用フェーズ）。
詳細設計＝「**どう作るか**」。要件定義（petrichor）で確定した「何を作るか」を、**採用スタックに対して**実装可能な形に落とす。
**設計の散文は `docs_lang` に従う**（petrichor spec ヘッダ由来、なければ**聞かずに ja**。
公開系 — OSS README 等 — のときだけ en / ja+en を提案；
`ja+en` は正本＋レンダリング訳 — 訳は正本から再生成し、直接編集しない）。
コード成果物（Lint 設定・OpenAPI・IaC 等）は repo の流儀に従う。

## スコープ境界

- 扱うのは「どう作るか」= 実装に必要な設計と開発環境の確立まで。
- 「何を作るか」（要件定義）は **petrichor**。重複しない。Claude Code の `.claude/` 設定は本スキル squall の最終ステップで記録する。
- **出力は repo 内**（コードと一緒に versioning する）。

## 入力

- 要件定義 spec（`SPEC.md` or `<shared-root>/<project>/petrichor-plan/00-overview.md`）
  — 特に 機能要件一覧 / 画面定義 / **データ設計** / システム構成図(To-Be)。
- 既存コード・採用スタック・ライブラリ・フレームワークの作法。

## 横断原則

- **探索ファースト**: spec とコードから分かることは聞かない。
- **コアは妥協しない**（DB のリレーション等、失敗時のダメージが大きい所）。それ以外は実装着手可能な粒度で十分。
- ツールで縛れる規約（命名・整形）は **Lint / formatter の設定**に落とす（散文だけにしない）。
- 全機能を細部まで設計しない。**複雑 / 再利用 / リスクの高い部分**に絞る。
- 該当しないセクションはスキップしてよい（理由を記録）。
- ゴールは「**実装着手可能な粒度**」。

## セクション（成果物）

各セクションは **目的 / 開始条件 / 終了条件(Done) / 主な項目 / レビュー観点** を持つ。依存順に進める。

### 1. 開発環境・規約

- 目的: 実装前に開発環境とルールを確立する（後回しにすると工数が膨らむ）。
- 主な項目:
  - **環境構築手順（README）**: 前提条件、セットアップ手順、依存関係。
    `docs_lang: ja+en` の場合は OSS の二枚看板（`README.md`＝正本側の言語 ＋ `README.ja.md` / 冒頭に相互リンク）。
    訳側は正本から再生成する運用（直接編集しない旨を冒頭に一行）。
  - **タスク管理手順**: タスクの書き方 / 報告方法 / 進捗管理（例: デイリーに出られない時の報告ルール）。
  - **コーディング規約・静的解析**: 命名規則 / 変数名 / ディレクトリ構造 / ファイルの責務。細かい規則は Lint で潰す（＝Lint 設定に落とす）。
- 終了条件: 新規参加者が README だけで環境を立てられる / 規約が Lint・formatter 設定もしくは文書で明文化されている。
- レビュー観点: ルール変更時に更新される運用になっているか / 「どこに何を作るか」が決まっているか。

### 2. DB 詳細設計

- 目的: 要件定義のデータ設計（論理）を物理スキーマに落とす。
- 開始条件: 要件定義の データ項目定義書 / ER 図がある。
- 主な項目: テーブル定義（物理名 / 型 / 桁 / 制約 / PK・FK / index）、リレーション、マスタ / 履歴の分離。
- 終了条件: コアなリレーションが確定し破綻しない / マイグレーションが書ける粒度。
- レビュー観点: 拡張を見越した分離（マスタ・履歴）/ 画面都合のテーブル乱造・JSON 逃げ・履歴の後付けは NG。

### 3. モジュール / 処理設計

- 目的: 分かりづらい機能・再利用部品（認証モジュール等）の内部設計。
- 主な項目: 処理概要 / インターフェース / 処理フロー・ロジック / 関連 DB。処理フローはフローチャートで図示。
- 開始条件: 機能要件一覧が確定している。
- 終了条件: 実装者が迷わず書ける / 例外・状態遷移が定義済み。
- レビュー観点: 全機能ではなく、複雑・再利用・リスクの高い部分に絞れているか。

### 4. インターフェース設計

- 目的: コンポーネント間 / 外部システムとの契約を確定する。
- 主な項目:
  - **API 定義書**: OpenAPI（yaml）でエンドポイント / リクエスト / レスポンス / エラーを記述。
  - **シーケンス図**: DB・外部システム・サービス間連携が絡む処理の流れ（マイクロサービス等で有効）。
- 終了条件: フロント / バックが独立して実装に入れる契約になっている。
- レビュー観点: エラー・異常系のレスポンスが定義済みか。

### 5. インフラ詳細

- 目的: 要件定義のシステム構成図(To-Be)を、再現可能な構成値に落とす。
- 主な項目: ミドルウェアのバージョン / 設定値 / 権限 / IaC 方針。
- 開始条件: システム構成図(To-Be)・非機能要件が確定している。
- 終了条件: 構築担当が環境を再現できる / 設定値が文書もしくは IaC 化されている。
- レビュー観点: 本番だけでなく STG など必要な環境が網羅されているか。

### 6. テスト設計（EARS → 自動テスト）

- 目的: spec の**受け入れ条件（EARS 文）を自動テストに落とす設計**を決める。手動シナリオ（forecast）との分担線もここで引く。
- 開始条件: 機能要件一覧（受け入れ条件つき）とモジュール / IF 設計が一通りある。
- 主な項目:
  - **フレームワーク・置き場所・実行方法**（`check` の test ステップから回る形にする）。
  - **命名規約 — テスト名に機能 ID を刻む**（例: `F-01: ログイン済みのとき、保存ボタンで下書きが保存される`）。
    ID が刻まれていると「テストのない v1 機能」を weathering が機械的に検出できる。
  - **EARS → テストケースのマッピング**: WHEN 節 = Arrange/Act、SHALL 節 = Assert、IF 節 = 例外系ケース。
    受け入れ条件 1 行につき最低 1 ケース。
  - **自動 / 手動の分担**: 自動化しない項目（視覚確認・外部サービス実挙動など）は「手動のみ」と明示し、
    forecast のシナリオ側で拾う（黙って未カバーにしない）。
- 終了条件: v1 の全機能 ID が「自動テストあり」か「手動のみ（理由つき）」のどちらかに割り付いている。
- レビュー観点: テストが**受け入れ条件の言い換え**になっているか（実装詳細のテストになっていないか）/
  ID トレースが取れるか / 例外系（IF 節）が漏れていないか。

### 7. タスク分解（tasks.md = ビルド台帳）

- 目的: 確定した設計を「**依存順・追跡可能な実装単位**」に分解し、ビルド中の**コーディネーション台帳**にする。マルチセッション / worktree 並列でも「何が残っているか・依存は何か・どこまで終わったか」が揮発しない。**プランと進捗を1ファイルにまとめて持つ**。
- 開始条件: 上記セクション（特に DB・モジュール・IF）が実装着手可能な粒度に達している。
- 主な項目（プラン部）: タスク ID（`T-001`…）/ タイトル / **対応する機能 ID・画面 ID**（要件側のトレーサビリティをそのまま延長）/ **依存タスク**（先行 `T-xxx`）/ **完了条件**（何ができたら done か＋検証手段。**spec に受け入れ条件があればそこから導出・引用**する — 新しい基準を発明しない。実装後の `verify` が照らすのも同じ基準）/ 規模感（任意）/ **状態**（todo / in-progress / done）。
- 進捗の持ち方: 各タスク行の**状態欄**を更新し、加えて末尾に `## 進捗ログ`（append-only。各行 `T-001 done (branch-x)` のように追記のみ）を置く。**追記中心にすることで並列 worktree の同時更新でもクロバーしにくく**、cross-worktree で「今どこまで」が単一の場所で見える。
- 終了条件（着工可否）: 全タスクに ID・依存・完了条件があり、依存グラフに循環がなく、**機能要件一覧の v1 全機能が最低 1 つのタスクに対応**している（落とし込み漏れがない。v2 / 保留 は入れない — 先送りしたスコープを先食いしない）。
- レビュー観点: 各タスクが独立して着手・検証できる粒度か / 依存が実際の技術依存を反映しているか（順序の都合で過剰に直列化していないか）/ 機能 ID → タスク ID のトレースが一本で説明できるか。
- スケール: **L2/L3 の実質的なビルドで作る。express lane / 一枚物の修正では作らない。**
- 置き場所: **repo の外** — `<shared-root>/<project>/tasks.md`（shared root はデフォルト `~/Documents/claude-shared`、`~/.claude/shared-dirs.json` でプロジェクト別上書き可 — global Handoff ルール。`feedback.md` の隣、Obsidian 可読、throwaway・コミットしない）。設計の耐久記録は設計ドキュメント＋コード自体が持つので、これは「今回どう順序立てて作るか」の一時足場。repo 外なので可変な進捗を自由に持ててよく、monsoon の「可変ステートをコミット物に置かない」原則とも衝突しない（そもそもコミットしない）。

## 完了条件

該当する全セクションが各「終了条件」を満たし、**実装に着手できる**状態。
加えて、着工前に一度だけ **成果物間の横断整合**を通しで確認する（spec ↔ design ↔ tasks。詳細は SKILL.md の Done）。
最後に**着工承認（GO）**を得る — 設計サマリ 1 画面（主要決定・スキップ・タスク数・リスク）を提示し、GO を待つ（L1 / 軽微はスキップ。SKILL.md の Done ゲート 3）。
その後、本スキルの最終ステップで規約・設定を `.claude/` に記録し、通常ループでビルド（ビルド規律は ambient＝global CLAUDE.md）→ `monsoon` が巡回の次ステップを振る。
